02 — Background

What Is a SIEM? Why Splunk?

Every time a user logs in, a process runs, a firewall blocks a connection, or a file is modified — the operating system writes a log event. On a network with hundreds of machines, millions of these events are generated every day. No human can read them all.

A SIEM solves this by:

1
Collecting
Pulling log events from every endpoint, server, firewall, and cloud service into one central platform.
2
Normalising
Converting different log formats (Windows XML, Linux syslog, firewall CEF) into a consistent, queryable schema.
3
Correlating
Running detection rules that look across multiple log sources simultaneously — e.g. 5 failed logins followed by a successful one from the same IP = credential stuffing attack.
4
Alerting
Sending real-time notifications to the security team when a detection rule fires, with context about what triggered it.
Why Splunk specifically?
Splunk is the most widely used SIEM in enterprise environments — it appears in job postings more than any other SIEM tool. The free tier (500MB/day) is sufficient for a home lab. As a Splunk user at Essen Vision, building a personal Splunk project directly demonstrates real-world applicable skills to employers.

05 — Sysmon

Install & Configure Sysmon for Rich Telemetry

Default Windows Event Logs are extremely limited — they miss 80% of what attackers do. Sysmon (System Monitor) is a free Microsoft tool that captures process creation, network connections, file hashing, registry changes, and more — giving you the telemetry that actually catches attacks.

5.1 Download Sysmon and Config

1
Download Sysmon from Microsoft
Go to https://docs.microsoft.com/sysinternals/downloads/sysmon → download Sysmon.zip. Extract it to C:\Tools\Sysmon\
2
Download the SwiftOnSecurity config
This is the industry-standard Sysmon config file. Run in PowerShell:
Invoke-WebRequest -Uri "https://raw.githubusercontent.com/SwiftOnSecurity/sysmon-config/master/sysmonconfig-export.xml" -OutFile "C:\Tools\Sysmon\sysmonconfig.xml"
3
Install Sysmon with the config
Open PowerShell as Administrator and run:
cd C:\Tools\Sysmon
.\Sysmon64.exe -accepteula -i sysmonconfig.xml
4
Verify Sysmon is running
Get-Service Sysmon64 — should show Status: Running
Expected Result
System Monitor v15.x has started.

06 — Log Forwarding

Configure Splunk Universal Forwarder

The Splunk Universal Forwarder (UF) is a lightweight agent you install on your Windows VM. It ships logs from that machine to your Splunk server automatically. Think of it as the postman that collects log events and delivers them to Splunk.

6.1 Install the Universal Forwarder on the Windows VM

1
Download the Universal Forwarder
From https://www.splunk.com/en_us/download/universal-forwarder.html → download the Windows 64-bit MSI installer.
2
Install the UF
Run the MSI → set a deployment server address pointing to your Splunk server IP on port 8089 (e.g. 192.168.56.102:8089) → set a receiving port of 9997.

6.2 Configure inputs.conf to collect the right logs

Edit the inputs configuration file at C:\Program Files\SplunkUniversalForwarder\etc\system\local\inputs.conf:

# Windows Security Event Log — logins, account changes, privilege use
[WinEventLog://Security]
disabled = 0
index = windows
sourcetype = WinEventLog:Security

# Sysmon — rich process/network/file telemetry
[WinEventLog://Microsoft-Windows-Sysmon/Operational]
disabled = 0
index = sysmon
sourcetype = XmlWinEventLog:Microsoft-Windows-Sysmon/Operational
renderXml = true

# Windows System Log
[WinEventLog://System]
disabled = 0
index = windows
sourcetype = WinEventLog:System

# Windows Application Log
[WinEventLog://Application]
disabled = 0
index = windows
sourcetype = WinEventLog:Application

6.3 Configure outputs.conf to point to Splunk

Edit C:\Program Files\SplunkUniversalForwarder\etc\system\local\outputs.conf:

[tcpout]
defaultGroup = splunk_server

[tcpout:splunk_server]
server = 192.168.56.102:9997   # Replace with your Splunk server IP

6.4 Enable receiving on the Splunk server

# On your Splunk server — enable receiving on port 9997
/opt/splunk/bin/splunk enable listen 9997 -auth admin:yourpassword

# Restart both the Forwarder and Splunk to apply changes
# On Windows VM:
Restart-Service SplunkForwarder

# On Splunk server:
sudo /opt/splunk/bin/splunk restart

6.5 Verify data is flowing into Splunk

In Splunk Web → go to Search & Reporting → type:

index=windows OR index=sysmon | head 20
Expected Result
20 events appear in the results list showing Windows and Sysmon log entries. You will see fields like host, sourcetype, EventCode, and _time. ✅

07 — Attack Simulation

Simulate Attack Scenarios (Generate Telemetry)

With Splunk receiving logs, you now need to generate real attack events so you have something meaningful to detect. All attacks are run from your Kali Linux VM against the Windows VM. Both must be on the same Host-Only network (from Project 01).

Attack A: SSH / RDP Brute Force with Hydra

# From Kali — brute-force RDP login attempts against Windows VM
hydra -l Administrator -P /usr/share/wordlists/rockyou.txt rdp://192.168.56.103

# Alternative: brute-force SMB
hydra -l administrator -P /usr/share/wordlists/rockyou.txt smb://192.168.56.103

This generates Windows Security Event ID 4625 (failed login) events in Splunk — dozens per second during the attack.

📸 Let Hydra run for 60 seconds then stop it. Note the time — you'll search for these events in Splunk.

Attack B: Network Port Scan with Nmap

# From Kali — aggressive scan of Windows VM
nmap -sV -sC -O 192.168.56.103

# This generates Sysmon Event ID 3 (network connection) events
# as Windows processes respond to the scan probes

Attack C: Lateral Movement Simulation

# From Kali — attempt connections to multiple Windows services
# to simulate lateral movement behaviour
for port in 445 135 3389 5985; do
  nc -zv 192.168.56.103 $port &
done

Attack D: Suspicious Process Execution

On the Windows VM itself, simulate a suspicious process chain (the kind malware uses):

# Open PowerShell on Windows VM and run these commands
# They simulate what ransomware and RATs commonly do

# Simulate encoded PowerShell (malware evasion technique)
powershell -enc VwByAGkAdABlAC0ASABvAHMAdAAgACIASABlAGwAbABvACIA

# Simulate a process spawning cmd.exe (common malware behaviour)
Start-Process cmd.exe -ArgumentList "/c whoami" -WindowStyle Hidden

# Simulate directory traversal
Get-ChildItem C:\Users -Recurse -ErrorAction SilentlyContinue | Select FullName
📸 All these attacks generate rich Sysmon and Windows Security events in Splunk. Keep Splunk's Search open in one window while you run attacks in another.
08 — SPL Detection Rules

Write SPL Detection Queries

SPL (Splunk Processing Language) is Splunk's query language. These are your detection rules — each one finds a specific type of malicious behaviour in the log data.

Detection Rule 1 — Brute Force Login Attack

Detects when the same source IP attempts 5 or more failed logins within a 5-minute window.

SPL Rule 01 Brute Force Detection · EventCode 4625 
# Detect brute force: 5+ failed logins in 5 minutes from same source
index=windows EventCode=4625
| bucket _time span=5m
| stats count by _time, src_ip, Account_Name, ComputerName
| where count >= 5
| eval severity="HIGH", alert_type="BRUTE_FORCE_DETECTED"
| sort -count
| table _time, src_ip, Account_Name, ComputerName, count, severity, alert_type

Detection Rule 2 — Lateral Movement (Unusual Admin Logins)

Detects an account logging into more than 3 different machines via network logon — a strong indicator of lateral movement.

SPL Rule 02 Lateral Movement · EventCode 4624 LogonType 3 
# Detect lateral movement — account logging into 3+ different hosts
index=windows EventCode=4624 Logon_Type=3
| stats dc(ComputerName) as host_count, values(ComputerName) as hosts by Account_Name, src_ip
| where host_count >= 3
| eval severity="CRITICAL", alert_type="LATERAL_MOVEMENT"
| sort -host_count
| table Account_Name, src_ip, host_count, hosts, severity, alert_type

Detection Rule 3 — Suspicious PowerShell Execution

Detects encoded PowerShell commands — a technique commonly used by malware to evade simple signature detection.

SPL Rule 03 Suspicious PowerShell · Sysmon Event ID 1 
# Detect encoded PowerShell and suspicious command-line flags
index=sysmon EventCode=1
  (CommandLine="*-enc*" OR CommandLine="*-EncodedCommand*"
   OR CommandLine="*-WindowStyle Hidden*"
   OR CommandLine="*IEX*" OR CommandLine="*Invoke-Expression*"
   OR CommandLine="*DownloadString*")
| eval severity=case(
    match(CommandLine,"(-enc|-EncodedCommand)"), "HIGH",
    match(CommandLine,"(IEX|Invoke-Expression|DownloadString)"), "CRITICAL",
    true(), "MEDIUM"
  )
| stats count by _time, Computer, User, CommandLine, ParentCommandLine, severity
| sort -_time
| table _time, Computer, User, CommandLine, ParentCommandLine, count, severity

Detection Rule 4 — Port Scan Detection

Detects when Sysmon logs show a single external IP making connections to 10+ different ports in under a minute.

SPL Rule 04 Port Scan Detection · Sysmon Event ID 3 
# Detect port scanning — single source connecting to 10+ ports in 60 seconds
index=sysmon EventCode=3
| bucket _time span=1m
| stats dc(DestinationPort) as port_count, values(DestinationPort) as ports by _time, SourceIp, Computer
| where port_count >= 10
| eval severity="HIGH", alert_type="PORT_SCAN"
| sort -port_count
| table _time, SourceIp, Computer, port_count, ports, severity, alert_type

Detection Rule 5 — Suspicious File Created in Temp Folder

Malware almost always writes its payload to a temp folder first. This rule catches any new executable or script written to Windows temp directories.

SPL Rule 05 Malware Drop Detection · Sysmon Event ID 11 
# Detect executable / script files written to temp directories
index=sysmon EventCode=11
  (TargetFilename="*\\Temp\\*.exe" OR TargetFilename="*\\Temp\\*.ps1"
   OR TargetFilename="*\\Temp\\*.bat" OR TargetFilename="*\\Temp\\*.vbs"
   OR TargetFilename="*\\AppData\\Local\\Temp\\*.exe"
   OR TargetFilename="*\\Downloads\\*.exe")
| eval severity="HIGH", alert_type="SUSPICIOUS_FILE_DROP"
| stats count by _time, Computer, User, TargetFilename, Image, severity
| sort -_time
| table _time, Computer, User, TargetFilename, Image, count, severity

11 — Incident Report

Incident Report — Full Detection Walkthrough

This is a complete end-to-end incident investigation that you document as evidence of your skills. Follow these steps exactly and record every output.

Phase 1 — The Attack (T = 0:00)

From Kali Linux, launch the brute force attack against the Windows VM:

# Record the exact start time — you will need it for the SIEM query
echo "Attack started at: $(date)"

# Launch Hydra brute force against RDP
hydra -l Administrator -P /usr/share/wordlists/rockyou.txt rdp://192.168.56.103 -t 4 -V

# Let it run for 2 minutes then stop with Ctrl+C
echo "Attack stopped at: $(date)"

Phase 2 — Detection in Splunk (T = 2:00)

In Splunk Search, run the following to find the attack:

# Find all failed login attempts in the last 10 minutes
index=windows EventCode=4625 earliest=-10m
| stats count as failed_attempts, values(Account_Name) as targeted_accounts
  by src_ip, ComputerName
| where failed_attempts > 10
| sort -failed_attempts
📸 Screenshot the results showing the attacker IP (192.168.56.102), the target machine, and the count of failed attempts.

Phase 3 — Investigation (T = 3:00)

Pivot from the initial finding to understand the full scope of the attack:

# Timeline of all events from attacker IP in last 30 minutes
index=windows OR index=sysmon earliest=-30m src_ip=192.168.56.102
| table _time, index, EventCode, Account_Name, ComputerName, CommandLine, DestinationIp
| sort _time

# Check if attacker ever successfully logged in (EventCode 4624)
index=windows EventCode=4624 earliest=-30m
  (src_ip=192.168.56.102 OR Account_Name=Administrator)
| table _time, Account_Name, src_ip, Logon_Type, ComputerName
