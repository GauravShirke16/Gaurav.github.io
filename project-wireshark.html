01 — Overview

Project Overview & Objectives

Wireshark is the world's most-used network protocol analyser. Every packet that travels across a network leaves evidence — source IP, destination IP, protocol, payload, timing. Wireshark reads that evidence. Security engineers use it to inspect raw traffic, identify attack patterns, find credential theft, and conduct post-incident forensics after a breach.

This project captures real attack traffic from the home lab — Kali Linux attacking Metasploitable — analyses the packets using display filters, extracts Indicators of Compromise (IOCs), maps each attack technique to MITRE ATT&CK, and produces a professional threat hunting report with SIEM detection rules derived from the packet evidence.

02 — Key Concepts

What Does Wireshark See? Network Forensics Concepts

Before capturing packets, you need to understand the OSI model layers that Wireshark works across and what evidence each layer leaves behind.

LayerProtocol ExamplesWhat Attackers Leave Behind
Layer 2 — Data LinkEthernet, ARPARP spoofing (duplicate MAC/IP), MAC flooding
Layer 3 — NetworkIP, ICMPPort scan SYN floods, ICMP ping sweeps, fragmentation attacks
Layer 4 — TransportTCP, UDPHalf-open connections (port scanning), unusual port usage
Layer 7 — ApplicationHTTP, FTP, DNS, SMBCleartext credentials (FTP/HTTP), C2 beacons, data exfiltration
The "Follow TCP Stream" Killer Feature
Wireshark's Follow TCP Stream (right-click any TCP packet → Follow → TCP Stream) reassembles the entire conversation between two endpoints into human-readable text. This is how you extract cleartext FTP passwords, HTTP POST bodies, and plaintext command output from Telnet sessions — all visible in a single view.

04 — Setup

Install Wireshark & Configure Capture Interface

1
Download and install Wireshark on Windows
Go to https://www.wireshark.org/download.html → download Windows x64 Installer. During installation, tick "Install Npcap" — this is required for packet capture on Windows.
2
Select the correct capture interface
Open Wireshark → Welcome screen shows all network interfaces with live traffic sparklines. Select VirtualBox Host-Only Ethernet Adapter (this is the 192.168.56.0/24 lab network).
3
Set a capture filter (optional)
Before starting capture, click the capture filter bar and type: net 192.168.56.0/24 — this captures only lab network traffic and keeps file sizes manageable.
4
Configure capture file auto-save
Capture → Options → Output tab → tick "Save to file" → name: lab-capture-%Y%m%d-%H%M%S.pcap. Set ring buffer at 100MB per file to prevent huge files.
5
Start capture — blue shark fin button
Click the blue shark fin button at top-left or press Ctrl+E. Packets immediately begin streaming in. The interface counter shows packets per second.
Expected Result
Wireshark shows the live packet list streaming in. The interface statistics bar at the bottom shows captured packet count incrementing. Colour-coded rows: blue = SYN/TCP, green = normal TCP, yellow = DNS. ✅

06 — Attack Capture

Capture 2 — Attack Traffic (Nmap + Metasploit)

The second capture runs while actively attacking Metasploitable from Kali. This generates real attack telemetry: SYN floods from port scanning, Metasploit exploit packets, FTP credential transmission, and post-exploitation traffic.

1
Start Wireshark capture on Kali
sudo tshark -i eth1 -w attack-traffic.pcap
Or start Wireshark GUI on eth1 interface.
2
Run Nmap port scan (generates SYN flood)
nmap -sS -p 1-65535 192.168.56.101
This sends 65,535 SYN packets — visible in Wireshark as thousands of TCP SYN packets in seconds.
3
Run FTP login attempt (generates cleartext creds)
ftp 192.168.56.101 then login as msfadmin with password msfadmin — both username and password appear in plaintext in the packet capture.
4
Run Metasploit vsftpd exploit (generates C2 traffic)
Launch the vsftpd_234_backdoor exploit from Metasploit. The exploit traffic, command and control packets, and shell commands all appear in Wireshark.
5
Stop capture and save as attack-traffic.pcap
Stop tshark with Ctrl+C or stop Wireshark. The file contains all attack packets for analysis.
07 — Display Filters

Analyse with Wireshark Display Filters

Display filters narrow down thousands of packets to exactly the traffic you care about. Unlike capture filters (which filter during recording), display filters work on already-captured PCAPs without discarding any data.

Right-click → Follow TCP Stream
After identifying a suspicious conversation with a display filter, right-click any packet in that stream → Follow → TCP Stream. 09 — ARP Spoofing

ARP Spoofing Detection

ARP spoofing (also called ARP poisoning) is the foundation of man-in-the-middle attacks on local networks. An attacker sends forged ARP replies claiming their MAC address is the gateway IP — redirecting all network traffic through the attacker's machine.

# SIMULATE ARP SPOOFING (from Kali, against lab network only)
sudo arpspoof -i eth1 -t 192.168.56.101 192.168.56.1

# In a separate terminal — enable IP forwarding (so traffic still flows)
echo 1 | sudo tee /proc/sys/net/ipv4/ip_forward

# DETECT in Wireshark — apply this display filter:
arp.duplicate-address-detected

# Also detect with tshark:
tshark -r attack-traffic.pcap -Y "arp" \
  -T fields -e arp.src.hw_mac -e arp.src.proto_ipv4 | \
  sort | uniq -c | sort -rn
# If same IP appears with two different MAC addresses = ARP spoofing
⚠ Simulating ARP spoofing in the home lab only
Only run arpspoof against VMs in your own isolated lab network. ARP spoofing on a real network disrupts connectivity for all users on the subnet and is an attack — not just a scan. In the lab, it is safe because the host-only network is completely isolated.
What ARP Spoofing Looks Like in Wireshark
In the packet list you will see: 192.168.56.1 is at 08:00:27:aa:bb:cc followed immediately by 192.168.56.1 is at 08:00:27:ff:ee:dd — the same IP claiming two different MAC addresses. Wireshark automatically colours these rows red and marks them as "duplicate-address-detected".

11 — SIEM Detection Rules

Write SIEM Detection Rules (Splunk SPL)

The final analytical output is converting PCAP findings into Splunk SPL detection rules that can be deployed in a live SIEM. Each rule is derived directly from the packet-level evidence — making them highly targeted and low false-positive.

Rule 1 — Detect Port Scanning (SYN Flood)

# Derived from: 847 SYN-only packets to 60+ ports in 14 seconds
# MITRE: T1046 — Network Service Discovery
index=network sourcetype=firewall action=allow
| where tcp_flags="S"
| bucket _time span=10s
| stats dc(dest_port) as port_count by _time, src_ip, dest_ip
| where port_count >= 15
| eval alert="PORT_SCAN_DETECTED", severity="HIGH"
| table _time, src_ip, dest_ip, port_count, alert, severity

Rule 2 — Detect Cleartext FTP Credentials

# Derived from: FTP PASS command visible in packet 148
# MITRE: T1040 — Network Sniffing
index=network sourcetype=ftp
| where ftp_command="PASS" OR ftp_command="USER"
| eval alert="CLEARTEXT_CREDENTIAL_TRANSMITTED", severity="CRITICAL"
| table _time, src_ip, dest_ip, ftp_command, ftp_argument, alert

Rule 3 — Detect Metasploit C2 on Port 4444

# Derived from: bidirectional TCP session on port 4444 post-exploit
# MITRE: T1059 — Command and Scripting Interpreter
index=network sourcetype=netflow
| where dest_port=4444 OR src_port=4444
| stats count by src_ip, dest_ip, dest_port
| where count > 5
| eval alert="METASPLOIT_C2_DETECTED", severity="CRITICAL"
| table _time, src_ip, dest_ip, dest_port, count, alert

Rule 4 — Detect vsftpd Backdoor Port 6200

# Derived from: SYN to port 6200 (vsftpd 2.3.4 backdoor trigger)
# MITRE: T1203 — Exploitation for Client Execution
index=network sourcetype=firewall
| where dest_port=6200
| eval alert="VSFTPD_BACKDOOR_TRIGGERED", severity="CRITICAL"
| table _time, src_ip, dest_ip, alert, severity

Rule 5 — Detect ARP Spoofing (Duplicate IPs)

# Derived from: duplicate ARP announcement for same IP (different MAC)
# MITRE: T1557.002 — ARP Cache Poisoning
index=network sourcetype=arp
| stats dc(src_mac) as mac_count, values(src_mac) as mac_list by src_ip
| where mac_count >= 2
| eval alert="ARP_SPOOFING_DETECTED", severity="HIGH"
| table _time, src_ip, mac_count, mac_list, alert, severity
