01 — Overview

Project Overview & Why Write-ups Matter

TryHackMe is a browser-based cybersecurity training platform where you complete hands-on "rooms" — guided challenges covering everything from Linux fundamentals to advanced Active Directory exploitation. Each room challenges you to hack into a real (virtualised) machine, capture flags, and demonstrate understanding of the technique.

Why write-ups are the key differentiator: anyone can complete a room. A write-up proves you understood the technique well enough to explain it, document it professionally, and teach it to someone else. Hiring managers and recruiters actively look for candidate write-ups on GitHub and personal blogs — they are the single best evidence of hands-on skill beyond a CV line.

TryHackMe Platform — Getting Started

1
Create a free TryHackMe account

Start with the Pre-Security learning path
If new to cybersecurity, complete the Pre-Security path first — it covers networking, Linux, and web fundamentals. Intermediate users can skip to individual rooms directly.
5
Subscribe for full access (optional)
The free tier gives access to ~60% of rooms. A monthly subscription unlocks all 500+ rooms and learning paths. For this project, free-tier rooms are prioritised first.

03 — Lab Setup

Lab Setup — Kali Linux & OpenVPN Connection

1
Download the TryHackMe OpenVPN config
TryHackMe profile → Access → Download My VPN Config → choose your region (EU / US) → download gauravshirke16.ovpn
2
Connect from Kali Linux
sudo openvpn --config gauravshirke16.ovpn
When you see "Initialization Sequence Completed" the VPN is active. Keep this terminal open throughout your session.
3
Verify the VPN connection
ip addr show tun0 — you should see a 10.x.x.x IP on the tun0 interface. This is your TryHackMe network address.
4
Start each room and note the target IP
When you deploy a machine in a room, TryHackMe assigns it a target IP (e.g. 10.10.x.x). Use this IP for all scanning and exploitation commands.
# Connect to TryHackMe VPN from Kali
sudo openvpn --config gauravshirke16.ovpn

# In a separate terminal — verify connection
ip addr show tun0
ping -c 3 10.10.10.10  # Replace with room target IP

# Standard first commands in every room
export TARGET=10.10.x.x  # Set target IP as variable
nmap -sV -sC -O $TARGET -oN nmap_initial.txt
Workspace folder structure for write-ups
Create a folder for each room: mkdir -p ~/thm/room-name/{nmap,exploits,loot,writeup}
Keep all command outputs saved to files — these become your write-up evidence automatically.
04 — Write-up Format

Write-up Format & Documentation Standard

Every write-up follows the same professional structure — mirroring how a penetration testing report is written. This consistency makes the collection look serious to employers.

# TryHackMe Write-up Template — room-name.md

## Room: [Room Name]
**Platform:** TryHackMe  
**Difficulty:** Easy / Medium / Hard  
**Category:** Privilege Escalation / Web / Network / etc.  
**Date completed:** DD MMM YYYY  
**Points earned:** XXX  
**MITRE ATT&CK techniques:** T1548.001, T1059, etc.

---

## Objective
[One paragraph — what the room is about and what you need to achieve]

## Enumeration
[Nmap scan, service discovery, web enumeration — commands + output]

## Exploitation
[How initial access was gained — commands, payloads, screenshots]

## Privilege Escalation
[How you went from low-privilege user to root/SYSTEM — step by step]

## Flags Captured
- User flag: `THM{xxxxxxxxxxxxxxxxxxxxxxxxxxxxxx}`
- Root flag: `THM{xxxxxxxxxxxxxxxxxxxxxxxxxxxxxx}`

## Tools Used
[Nmap, Gobuster, Metasploit, linpeas, winPEAS, etc.]

## Key Lesson Learned
[One clear technical insight from this room — what you will remember]

## Remediation (Defender's View)
[How the vulnerability could be fixed in a real environment]
Why include Remediation in CTF write-ups?
Most CTF write-ups only cover the attack side. Adding a remediation section demonstrates you think like a defender — not just an attacker. This is the key differentiator that makes your write-ups stand out to security engineering employers.

Room 01 — Linux Privilege Escalation

THM · 01 Linux Privilege Escalation Easy

Learn the fundamentals of Linux privilege escalation: exploiting SUID binaries, sudo misconfigurations, cron jobs running as root, writable PATH entries, and kernel exploits to escalate from a low-privilege user to full root access.

MITRE ATT&CK Techniques
T1548.001 — Setuid/Setgid T1053.003 — Cron Job T1574.007 — Path Interception T1068 — Exploitation for PrivEsc
Flags Captured
🚩User flag: THM{a_userflag_example_hash_here}
🏆Root flag: THM{rootflag_privilege_escalated}
Key Lesson Learned
Always run sudo -l and find / -perm -4000 2>/dev/null as the first commands after gaining a shell. SUID binaries and sudo misconfigurations are by far the most common PrivEsc vector on real systems — including misconfigured enterprise Linux servers.

Room 02 — OWASP Top 10 (Web Security)

THM · 02 OWASP Top 10 — 2021 Easy

Hands-on exploitation of the ten most critical web application vulnerabilities: Broken Access Control, Cryptographic Failures, Injection (SQLi), Insecure Design, Security Misconfiguration, SSRF, and more — each with a dedicated lab.

MITRE ATT&CK Techniques
T1190 — Exploit Public-Facing App T1059.007 — JavaScript T1552.001 — Credentials in Files T1083 — File and Directory Discovery
Key Commands
# SQL Injection — bypass authentication
' OR 1=1 --
admin'--

# Command Injection test
; whoami
&& cat /etc/passwd

# Directory traversal
../../../etc/passwd
..%2F..%2F..%2Fetc%2Fpasswd

# Gobuster — web directory bruteforce
gobuster dir -u http://10.10.x.x -w /usr/share/wordlists/dirb/common.txt
Key Lesson Learned
SQL Injection is still #1 in real production systems in 2025. Always test login forms with ' OR 1=1 -- before moving to automated tools. IDOR (Insecure Direct Object Reference) — changing ?id=1 to ?id=2 — is responsible for some of the most high-profile data breaches in recent years.

08 — Write-up #4

Room 04 — Nmap — Network Scanning Mastery

THM · 04 Nmap — The Complete Guide Easy

Master every aspect of Nmap: host discovery, TCP/UDP port scanning, service version detection, OS fingerprinting, NSE scripting for vulnerability detection, and output formats for report documentation.

Key Commands Learned
# Host discovery — find live hosts
nmap -sn 10.10.0.0/24

# SYN scan (fastest, stealthy)
sudo nmap -sS 10.10.44.182

# Full service + version + OS + scripts + save output
nmap -sV -sC -O -A 10.10.44.182 -oN full_scan.txt

# Specific port range
nmap -p 1-65535 10.10.44.182

# UDP scan (slow — target key ports)
sudo nmap -sU -p 53,67,68,69,123,161,500 10.10.44.182

# NSE vulnerability scripts
nmap --script vuln 10.10.44.182
nmap --script=smb-vuln* -p 445 10.10.44.182

# Aggressive timing (faster, noisier)
nmap -T4 -A 10.10.44.182
Key Lesson Learned
Nmap scan type selection matters enormously in real engagements. A SYN scan (-sS) is half-open and does not complete the TCP handshake — it is less likely to be logged. A Connect scan (-sT) completes the handshake and appears in application logs. In a real pentest, always discuss which scan type is approved with the client before running.

THM · 05 SOC Level 1 — Splunk & SIEM Fundamentals Medium

The full SOC analyst workflow: ingesting logs into Splunk, writing SPL detection queries, triaging alerts from an IDS/IPS, classifying incidents by severity, and escalating appropriately — directly complementing Project 02 (Splunk SIEM).

Key SPL Queries from This Room
# Find all failed login attempts (brute force)
index=windowslogs EventCode=4625
| stats count by IpAddress, UserName
| where count > 5
| sort -count

# Detect lateral movement
index=windowslogs EventCode=4624 LogonType=3
| stats dc(ComputerName) as hosts by UserName
| where hosts > 2

# Search for powershell encoded commands
index=windowslogs EventCode=4688
| search CommandLine="*-enc*" OR CommandLine="*-EncodedCommand*"
| table _time, UserName, CommandLine, ComputerName
Key Lesson Learned
The hardest part of SOC work is not writing the SPL query — it is knowing what to look for. This room taught me the specific Event IDs that matter: 4625 (failed login), 4624 (successful login), 4688 (new process), 4698 (scheduled task). Memorising these is more valuable than any tool certification.
